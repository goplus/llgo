Kept source optimizations: compact huge `newlibesp.go` and `compiler_rt.go` file lists from many `filepath.Join` expressions into raw string lists, improving median to 10301ms. - For future compile-speed work, investigate dependency/package-graph reductions and LLGo's own build pipeline parallelism. Kept fingerprint/cache-stage cleanups: `collectFingerprint` now hashes the manifest string it already built instead of calling `m.Fingerprint()` and rebuilding YAML; synthetic benchmark improved from 126488ns/op and 1311 allocs/op to 69680ns/op and 676 allocs/op. `saveToCache` now skips decode/re-marshal when metadata is empty and writes the already-built manifest unchanged; synthetic empty-metadata benchmark improved from 314572ns/op and 247 allocs/op to 295421ns/op and 42 allocs/op. `parseManifestMetadata` now avoids YAML unmarshal for modern manifests with no top-level metadata section while preserving YAML-with-metadata and legacy INI parsing; synthetic no-metadata benchmark improved from 50702ns/op and 905 allocs/op to 714ns/op and 1 alloc/op. For modern manifests with metadata, it now unmarshals only the top-level metadata section; synthetic metadata benchmark improved from 64467ns/op and 973 allocs/op to 7831ns/op and 104 allocs/op. `digestFilesWithOverlay` now skips `sort.Slice` for zero/one-file lists; synthetic single-file benchmark improved from 5368ns/op and 4 allocs/op to 1382ns/op and 3 allocs/op. Replacing its multi-file `sort.Slice` with `slices.SortFunc` was rejected because the eight-file benchmark only moved 10849ns->10501ns (~0.35µs) despite fewer allocs. The analogous `collectDependencyInputs` single-import sort guard was rejected despite a 104ns->71ns microbenchmark change because the absolute gain is too small for real build wall time. Returning the original slice from `sortDeps` for a single dependency was also rejected (7425ns->6854ns) because the absolute gain was ~0.57µs and introduced avoidable aliasing assumptions. Hoisting the `collectEnvInputs` env-var list was rejected (734ns->730ns, unchanged allocs). Avoiding the empty `p.OtherFiles` copy in `collectPackageInputs` was rejected (30717ns->30287ns, unchanged allocs). For non-empty cache metadata, `saveToCache` now marshals only the metadata block and inserts it into the already-built manifest, improving the synthetic metadata save benchmark from 428810ns/op and 1371 allocs/op to 317801ns/op and 102 allocs/op while preserving parsed metadata and metadata-before-deps placement. `fingerprintManifest` now avoids copying the manifest string before SHA-256 by using a read-only unsafe byte view; a ~23KB manifest benchmark improved from 9156ns/op and 24704 B/op to 7773ns/op and 128 B/op with an equivalence test against sha256. A follow-up that merely deferred `manifestMetadata` allocation in the empty save path saved one allocation but only moved wall time by ~0.1%, so it was rejected as noise-level churn. `buildAllPkgs` is sequential today; any package-level parallelization must be dependency-aware and guarded because LLVM/C ABI transformer/package generation may not be thread-safe. Safer external-process parallelism was implemented in `CompileGroup`: independent C/ASM files now compile concurrently (default `min(GOMAXPROCS,16)`, override `LLGO_COMPILE_JOBS`, per-file command logs opt-in via `LLGO_COMPILE_VERBOSE`). A 64-file local compile-group benchmark improved from 905ms sequential to 173ms default parallel; a 96-file tuning run improved from 241.5ms at 8 jobs to 164.8ms with the cap raised to 16. `CompileConfig.Compile` now shares one worker pool across archive groups and archives deterministically afterward; a synthetic 4-group/96-file benchmark improved from 233.5ms to 189.5ms while crosscompile tests and clean cmd/llgo go-build checks passed. Independent archive creation now also runs in parallel after compilation; an 8-group/128-file benchmark improved from 257.9ms to 215.7ms. Multi-group worker compiler lookup now uses a dense slice instead of a map; the 8-group/128-file benchmark improved from 222.4ms to 219.2ms, with a 9-run confirmation median of 215.8ms. The archive phase now precomputes the `llvm-ar` path once; the 8-group/128-file benchmark improved from 218.2ms to 213.3ms, with a 9-run confirmation median of 209.7ms. CompileGroup object files now use deterministic per-index paths inside the isolated temp dir instead of one `os.CreateTemp` per source; a 96-file benchmark improved from 167.1ms to 158.0ms. Simple `fmt` usage was removed from the CompileGroup hot package; a 96-file benchmark improved from 162.9ms to 159.7ms and clean cmd/llgo go-build remained in the recent-noise band (10430.6ms median). The common single-group parallel worker path was specialized to avoid per-task group-map lookups; a 128-file benchmark improved from 225.5ms to 208.8ms, with a 3-run confirmation median of 213.4ms. A direct single-group CompileGroup refactor was tested and rejected (216.1ms to 220.0ms). Changing archive mode from `llvm-ar rcs` to `qcs` was also rejected: isolated archive microbenchmark improved only ~1.5%, while the 8-group CompileConfig workload was slightly worse (209.7ms rcs/precomputed baseline vs 210.8ms qcs) and archive semantics churn was not justified. Preallocating the internal compile-task slice was also rejected (no allocation win visible in subprocess-dominated 8-group/128-file benchmark). Tracking the active group to avoid the `singleTaskGroup` scan was rejected too (128-file median 208.4ms baseline vs 211.8ms changed). Preallocating `archiveObjects` argv capacity showed only a 0.5ms/noise-level delta on the 8-group/128-file benchmark and was rejected as non-obvious code churn. Simple crosscompile archive-name `fmt.Sprintf` calls were replaced with string concatenation; a noisy clean go-build confirmation showed 10911.7ms baseline vs 10551.5/10573.8ms post-change, but earlier unrelated samples were ~10341ms so treat this as a small safe cleanup. Newlib ESP `joinFileList` now cleans the root once and uses prefix concatenation instead of per-entry `filepath.Join`, improving RISCV config construction from 112251ns/op to ~56504ns/op while preserving filepath.Join-equivalent output in tests. The same optimization in compiler-rt improved RISCV config construction from 12787ns/op to 6339ns/op with equivalent-output tests. Replacing newlib `strings.Fields` with a custom ASCII scanner was rejected: allocs dropped slightly (1424->1418) but wall time regressed (58607ns->65131ns). Sequential `clang.Cmd` wrapper reuse was tested and rejected (1316.6ms to 1350.2ms).

- For future compile-speed work, investigate dependency/package-graph reductions and LLGo's own build pipeline parallelism. `buildAllPkgs` is sequential today; any package-level parallelization must be dependency-aware and guarded because LLVM/C ABI transformer/package generation may not be thread-safe. Safer external-process parallelism was implemented in `CompileGroup`: independent C/ASM files now compile concurrently (default `min(GOMAXPROCS,16)`, override `LLGO_COMPILE_JOBS`, per-file command logs opt-in via `LLGO_COMPILE_VERBOSE`). A 64-file local compile-group benchmark improved from 905ms sequential to 173ms default parallel; a 96-file tuning run improved from 241.5ms at 8 jobs to 164.8ms with the cap raised to 16. `CompileConfig.Compile` now shares one worker pool across archive groups and archives deterministically afterward; a synthetic 4-group/96-file benchmark improved from 233.5ms to 189.5ms while crosscompile tests and clean cmd/llgo go-build checks passed. CompileGroup object files now use deterministic per-index paths inside the isolated temp dir instead of one `os.CreateTemp` per source; a 96-file benchmark improved from 167.1ms to 158.0ms. Simple `fmt` usage was removed from the CompileGroup hot package; a 96-file benchmark improved from 162.9ms to 159.7ms and clean cmd/llgo go-build remained in the recent-noise band (10430.6ms median). The common single-group parallel worker path was specialized to avoid per-task group-map lookups; a 128-file benchmark improved from 225.5ms to 208.8ms, with a 3-run confirmation median of 213.4ms. A direct single-group CompileGroup refactor was tested and rejected (216.1ms to 220.0ms). Simple crosscompile archive-name `fmt.Sprintf` calls were replaced with string concatenation; a noisy clean go-build confirmation showed 10911.7ms baseline vs 10551.5/10573.8ms post-change, but earlier unrelated samples were ~10341ms so treat this as a small safe cleanup. Sequential `clang.Cmd` wrapper reuse was tested and rejected (1316.6ms to 1350.2ms).
